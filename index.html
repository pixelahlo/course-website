📄 02

Як організована моя робота

Перш ніж починати будь-яку розробку — я пишу інструкцію для AI. Це найважливіший крок якому я навчилась. Без чіткої інструкції AI починає вигадувати і робити не те що потрібно.

Інструкція зберігається у файлі CLAUDE.md прямо в папці проекту. Там я описую:

  • що це за система і для чого
  • який стек технологій
  • правила безпеки
  • як AI повинен себе вести (питати перед тим як щось міняти, не вигадувати, показувати план перед кодом)
  • архітектурні рішення які вже прийняті
  • структуру бази даних і логіку бізнес-процесів

Цю інструкцію можна скласти разом з Claude в чаті — просто розповідаєш що хочеш побудувати, він задає питання і допомагає оформити все структуровано.

Як виглядає мій робочий процес

Я завжди тримаю два вікна відкритими одночасно:

  1. Cursor — там Claude Code пише і змінює файли, я бачу це в реальному часі
  2. Цей чат — тут я обговорюю логіку, задаю питання про те що незрозуміло, уточнюю вимоги

Claude Code запускається в терміналі всередині Cursor командою claude. Я пишу йому задачу — він читає файли проекту, пише код, запускає команди. Я бачу кожну дію і можу зупинити якщо щось іде не так.

⚡ 03

Скіли

Окрім базового Claude Code я встановила скіли — це набори інструкцій які перетворюють AI на спеціаліста в конкретній галузі. Кожен скіл це папка з файлом SKILL.md де описано як AI повинен поводитись в певній ситуації.

Різниця між скілами і агентами

  • Скіл — це інструкція-спеціалізація. AI читає її і починає думати як фахівець у цій галузі. Не потребує окремого запуску.
  • Агент — це окремий AI процес з власним контекстом який можна запустити паралельно для конкретної задачі.

Скіли які я використовую

🔥 grill-me
Коли пишу "grill me" перед описом задачі, AI починає задавати жорсткі питання про кожне рішення поки ми не дійдемо до повного розуміння.
✅ tdd
Test Driven Development. AI спочатку пише тест який перевіряє що код повинен робити, потім пише сам код.
🛡 setup-pre-commit
Автоматично перевіряє код перед кожним git push. Якщо є помилки — push не відбудеться.
📛 git-guardrails-claude-code
Блокує небезпечні git команди. Claude Code не може випадково видалити роботу.
🎨 frontend-design
AI створює красивий сучасний інтерфейс, не шаблонний.
📃 web-design-guidelines
Правила веб-дизайну для AI.
📦 04

Як встановити скіли

Скіли встановлюються однією командою в терміналі прямо в папці проекту:

terminalnpx skills@latest add mattpocock/skills/grill-me

При встановленні запитає:

  • для яких агентів → вибираєш Claude Code
  • scope → Project (буде в папці проекту і потрапить в git)
  • метод → Symlink (рекомендований)

Після цього колеги які зроблять git pull отримають ці скіли автоматично.

Варіант 2 — встановити локальний скіл

Якщо скіл це папка з файлом SKILL.md — просто кладеш її в проект:

your-project/ └── .claude/ └── skills/ └── skill-name/ └── SKILL.md

Де шукати скіли

Знайти скіли можна на claudemarketplaces.com або github.com. Я вибирала офіційні від Anthropic і перевірені від відомих розробників.

💻 05

Локальне тестування

Перед тим як відправити код на сервер — я завжди перевіряю локально. На моєму Mac запускається сервер:

npm run dev

І відкриваю в браузері http://localhost:5011 — бачу сайт як він виглядає. Ніхто інший цього не бачить, це тільки на моєму комп'ютері.

Якщо все добре — роблю push на GitHub, і через 30 секунд зміни автоматично з'являються на живому сайті.

📁 06

GitHub — налаштування

Реєструєшся на github.com, створюєш приватний репозиторій (сховище проекту).

Генеруєш Personal Access Token — це як пароль для терміналу:

GitHub → Settings → Developer settings → Personal access tokens → Generate new token (classic) → вибрати repo і workflow → зберегти одразу, більше не покажеться!

Підключаєш локальну папку до GitHub через термінал:

terminalgit remote add origin https://github.com/your-login/project-name.git git branch -M main git push -u origin main
🖥 07

Сервер

Отримуєш від хостинг-провайдера дані для підключення:

credentialsHost: server.example.app Login: billing Port: 42244 Path: /home/billing/web/site/public_html

Заходиш на сервер через термінал (не в Cursor — в окремому Terminal):

ssh -p 42244 billing@server.example.app

Що потрібно просити у адміна сервера

Звичайний користувач не має права встановлювати програми на сервері (sudo права). Тому деякі речі потрібно просити у адміна.

Наприклад, якщо потрібна база даних — після того як адмін зробить, отримуєш:

Host: localhost DB: billing_db User: billing_user Password: xxxxxxxxx

Ці дані додаєш у .env файл — він ніколи не потрапляє в GitHub. Файл .env з секретами (паролі, API ключі, дані бази даних) автоматично ігнорується через .gitignore.

🚀 08

Автодеплой

Налаштовується один раз. Після цього кожен git push автоматично оновлює живий сайт:

git push → GitHub → SSH → git pull → Перезапуск сервера

Для цього потрібно створити SSH ключ і налаштувати GitHub Actions — файл .github/workflows/deploy.yml. Claude Code допоможе зробити це крок за кроком.

☀ 09

Щоденна робота

Початок дня

git pull # get latest changes from colleagues

Якщо працюєш один, то немає сенсу робити.

Кінець дня

git add . git commit -m "what I did" git push # to GitHub → auto-deploy to server
👥 10

Командна робота

Кожен розробник працює у своїй гілці і не заважає іншим:

git checkout -b feature/name # create your branch # write your code... git push origin feature/name # push your branch

На GitHub створюється Pull Request — запит злити твою гілку в основну. Інші перевіряють і тільки тоді код потрапляє на продакшн.

main (live site) ├── feature/anna-duties ← Anna works on her part └── feature/ivan-labels ← Ivan works on his part
🗃 11

База даних на сервері

Перший раз після деплою — один раз:

npm run migrate # creates tables npm run seed # initial data (admin, references)

Після цього при кожному push база не чіпається — тільки код оновлюється. Нові таблиці — новий файл міграції.

📝 12

TASKS.md — як не втрачати контекст між сесіями

Claude Code не пам'ятає попередні сесії. Тільки бачить і читає файли. Файли .md — це його пам'ять та інструкції. Тому я веду ще файл TASKS.md прямо в папці проекту.

В кінці кожної сесії я прошу Claude оновити цей файл:

  • що зробили сьогодні (позначити як виконане)
  • що в процесі
  • що наступне
  • важливі рішення які прийняли

На початку наступної сесії Claude читає TASKS.md і одразу розуміє де ми зупинились. Не потрібно знову пояснювати контекст.

🔄 13

/compact & /clear

Під час довгої сесії Claude Code накопичує багато контексту в пам'яті. Коли сесія стає дуже довгою — AI починає "забувати" що було на початку або сповільнюється.

  • /compact — стискає всю історію сесії до короткого резюме. Звільняє місце в контексті але зберігає суть.
  • /clear — повністю очищує контекст і починає з нуля.

Після /compact обов'язково нагадуй AI прочитати CLAUDE.md і TASKS.md щоб він знову мав повний контекст проекту.

🔍 14

Як перевіряти роботу AI

AI може помилятись — особливо в бізнес-логіці яку він не до кінця розуміє. Що я роблю:

  • Прошу показати план перед кодом — так можна зупинити якщо іде не туди
  • Перевіряю числа — якщо AI каже "180 рядків оброблено" — відкриваю Excel і рахую сама
  • Тестую на реальних даних — у мене є тестові файли з відомим результатом, порівнюю
  • Читаю що він змінив — Cursor показує diff, я переглядаю перед тим як погодитись
  • Питаю "чому" — якщо рішення здається дивним, прошу пояснити логіку

Якщо щось виглядає підозріло — краще зупинитись і уточнити в чаті ніж потім виправляти.

📊 15

Status bar — зручний моніторинг сесії

Встановлюю один раз командою:

npx ccstatusline@latest

Потім прямо в чаті Claude Code прошу: "Add all available data to status line"

billingdesk main | Opus 4.6 (1M context) | [#———] 10% | $9.16 | 4776m14s

Показує в реальному часі:

  • назву проекту і гілку git
  • яка модель працює
  • скільки контексту використано (зелений <50%, жовтий 50-75%, червоний >75%)
  • вартість поточної сесії
  • тривалість сесії

Дуже допомагає розуміти коли час робити /compact і скільки коштує робота.
